# Yeoul v0.2 Product Architecture

This document defines the recommended Yeoul v0.2 product direction after the initial embedded core, policy pack, and CLI validation work.

It keeps Yeoul's existing core identity intact:

```text
Core는 AI를 모른다.
Agent-facing behavior lives outside the core.
```

The main change in v0.2 is not a new core model.
It is the addition of a stronger retrieval product layer around the existing temporal graph memory kernel.

## 1. Decision Summary

Yeoul v0.2 should be organized into four product-facing areas:

1. `Core`
2. `Retrieval`
3. `Agent`
4. `Evaluation`

The core remains the source of truth.
Retrieval, agent ergonomics, and evaluation become first-class product concerns, but remain derived from and constrained by the core.

## 2. Why this direction

Yeoul already has the strongest part of the system:

- local-first durable storage
- episode and fact primitives
- temporal validity
- provenance
- fact lifecycle operations such as supersede and retract

What is weaker today is the product layer around that model:

- search quality
- result construction for agent context
- session wake-up ergonomics
- evaluation discipline for quality, latency, and token cost

v0.2 exists to close that gap without turning Yeoul into an agent framework.

## 3. Governing constraints

These constraints remain fixed in v0.2:

- Ladybug-backed temporal graph storage remains the canonical store
- `Episode`, `Entity`, `Fact`, and `Source` remain the canonical memory objects
- provenance and lifecycle state remain core responsibilities
- retrieval summaries, projections, and community-level abstractions are derived artifacts, not source truth
- LLM-driven extraction, reranking, and answer behavior remain outside the core engine contract

## 4. Layered product model

Yeoul v0.2 should be understood as five nested layers:

1. storage
2. memory core
3. retrieval
4. agent interface
5. evaluation

The first two layers define truth.
The latter three layers make that truth usable in real workflows.

## 5. Core

The v0.2 core remains intentionally narrow.

### Responsibilities

- durable local persistence
- schema migration and storage integrity
- episodes as non-lossy source records
- entities and facts as semantic memory objects
- temporal semantics
- provenance traversal
- lifecycle operations such as assert, supersede, and retract

### Canonical model

The core model remains:

- `Episode`: durable source record
- `Entity`: resolved semantic node
- `Fact`: temporal claim with provenance and lifecycle
- `Source`: external or local origin reference

### Core must not do

- build agent prompts
- own wake-up summaries
- depend on embeddings or vector backends
- depend on a specific reranker
- treat derived summaries as truth

## 6. Retrieval

Retrieval is the main new product layer in v0.2.

It should follow a three-step contract:

1. `Search`
2. `Rerank`
3. `Constructor`

### 6.1 Search

Search should become explicitly hybrid.

The target shape is:

- `text lane`: lexical retrieval over projected records
- `vector lane`: optional semantic retrieval over projected records
- `graph lane`: neighborhood, path, support, and temporal adjacency traversal

The core graph is still the truth source, but search should operate over a derived projection optimized for recall and latency.

### 6.2 Search projection

Yeoul should add a derived search projection that can be rebuilt from the core.

The projection should include:

- searchable text for episodes, facts, and entities
- facet fields such as `space_id`, `group_id`, `source_kind`, `entity_type`, and `predicate`
- temporal fields such as `observed_at`, `valid_from`, `valid_to`, and active/inactive state
- graph hints such as `subject_id`, `object_id`, and supporting episode ids

This projection is a cache, not a canonical store.

See [ladybug-plus-rax.md](./ladybug-plus-rax.md) for the recommended dual-store integration model when `rax` is used as the retrieval substrate.

### 6.3 Rerank

Reranking should be explicit and bounded.

The default rerank stack should combine:

- lexical score
- semantic score when available
- graph proximity
- active fact preference
- temporal relevance
- provenance strength

Reranking must remain explainable enough that diagnostics can report why a result moved up or down.

### 6.4 Constructor

Search results should not stop at hit lists.

The constructor should transform ranked hits into agent-usable context blocks such as:

- active facts with valid ranges
- supporting entities
- source episodes
- concise provenance references

This is where Yeoul becomes easier to use in practice without moving answer behavior into the core.

### 6.5 Retrieval non-goals

- no derived summary becomes canonical truth
- no retrieval index may overwrite lifecycle history
- no hidden LLM dependency is required for search to function

## 7. Agent

The agent layer in v0.2 should remain outside the core but become much easier to use.

### 7.1 Main goal

The goal is not to embed agent logic inside Yeoul.
The goal is to expose better intent-level interfaces on top of the core and retrieval layers.

### 7.2 CLI-first product surface

The recommended v0.2 CLI additions are:

- `yeoul wake-up`
- `yeoul context`
- `yeoul decision prepare`
- `yeoul status explain`
- `yeoul context deep`

These commands should orchestrate existing primitives such as search, fact lookup, timeline, provenance, and neighborhood.

### 7.3 Wake-up model

Wake-up context should be layered:

- `L0`: identity and workspace policy
- `L1`: current durable state snapshot
- `L2`: task-scoped recall
- `L3`: deep retrieval and provenance drilldown

These layers are derived at read time and must not become authoritative stored memory.

### 7.4 Agent interface rules

- CLI and MCP should expose intent-level operations
- the policy pack should continue to drive behavior selection
- explicit lifecycle changes must still call core operations
- no agent adapter may silently rewrite history

## 8. Evaluation

Evaluation becomes a first-class product layer in v0.2.

The system should not optimize only for correctness in the abstract.
It must measure whether retrieval and construction are useful in practice.

### 8.1 Primary metrics

- retrieval quality
- response latency
- cold-start cost
- token budget of constructed context
- provenance coverage
- temporal correctness

### 8.2 Benchmark shape

Yeoul should evaluate:

- lexical retrieval
- hybrid retrieval
- temporal reasoning
- knowledge update handling
- multi-session continuity
- contradiction and supersede behavior

### 8.3 Evaluation rule

No retrieval upgrade should be accepted only because it improves recall if it materially degrades:

- provenance fidelity
- temporal correctness
- cold-start performance
- context construction cost

## 9. External influences and how Yeoul should use them

### 9.1 Zep / Graphiti

Useful ideas:

- episodic plus semantic memory split
- bi-temporal reasoning
- search, rerank, and constructor separation
- graph-aware retrieval

Things Yeoul should not copy directly:

- service-first architecture as the default product shape
- LLM-heavy construction inside the core
- treating community summaries as canonical truth

### 9.2 MemPalace

Useful ideas:

- productized retrieval UX
- wake-up and context loading ergonomics
- practical local agent tooling

Things Yeoul should not copy directly:

- retrieval-oriented structures replacing the core memory model
- conflating prompt summaries with durable memory

### 9.3 rax

Useful ideas:

- hybrid search execution boundaries
- vector lane structure
- explicit search-fusion contracts
- benchmark discipline

Things Yeoul should not copy directly:

- benchmark dataset pack assumptions as the product contract
- overly narrow text lane ranking

## 10. Recommended implementation order

1. define the search projection schema
2. add projection build, rebuild, and verification commands
3. introduce hybrid search planning and bounded rerank
4. add constructor-based result views
5. add CLI-first wake-up and context commands
6. formalize retrieval and temporal evaluation suites

## 11. v0.2 success criteria

Yeoul v0.2 is successful when:

- the core model remains stable and trustworthy
- retrieval quality is materially better than simple token overlap search
- common agent workflows no longer require manual primitive orchestration
- temporal and provenance correctness remain visible in outputs
- retrieval improvements are benchmarked against latency and context cost, not recall alone

## 12. One-line summary

Yeoul v0.2 should remain a temporal memory kernel at its core, while becoming a much stronger retrieval and agent-facing product around that kernel.
